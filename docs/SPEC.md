# Golf Weather — Specification

## Acceptance Criteria

### Location & Input
- [ ] ISC-1: App requests and receives GPS location permission on Android
- [ ] ISC-2: GPS location resolves to lat/lng within 5 seconds of permission grant
- [ ] ISC-3: Text search input accepts a place name and queries Open-Meteo geocoding API
- [ ] ISC-4: Geocoding returns lat/lng for the entered place name
- [ ] ISC-5: User can select from multiple geocoding results when a name is ambiguous
- [ ] ISC-6: Date picker allows selection from today through +15 days
- [ ] ISC-7: Date picker defaults to today's date on first load
- [ ] ISC-8: Round length picker accepts values: 3h, 3.5h, 4h, 4.5h, 5h (default: 4h)
- [ ] ISC-8b: If GPS permission is denied, app shows a clear fallback state with a text search input and a brief explanation (not a dead end)

### Weather Data
- [ ] ISC-9: App fetches Open-Meteo forecast for selected lat/lng and date
- [ ] ISC-10: Fetch includes hourly fields: apparent_temperature, precipitation, precipitation_probability, weather_code, wind_speed_10m, wind_gusts_10m
- [ ] ISC-11: Fetch includes daily fields: sunrise, sunset
- [ ] ISC-12: Units sent to API match user's selected unit preference (mph vs kph, °C vs °F)
- [ ] ISC-13: Loading indicator shown while API request is in flight
- [ ] ISC-14: Error state shown if API call fails (network down or API error)
- [ ] ISC-15: Empty state shown if no viable windows exist (not enough daylight / all excluded)

### Scoring Engine
- [ ] ISC-16: `scoreHour()` accepts wind speed, wind gust, precipitation, precipProb, and feelsLike — gusts contribute to the wind penalty score alongside sustained speed
- [ ] ISC-17: `scoreWindow()` computes `0.5 * mean(hourlyPenalties) + 0.5 * max(hourlyPenalties)` and clamps to 0-100
- [ ] ISC-18: `rankWindows()` generates all viable tee-off windows (step: 1h, rationale: coarser step = clearer choices for v1; 30-min step is v2) and returns them sorted by score descending
- [ ] ISC-19: Windows starting after `sunset - round_length - daylightBufferMins` are excluded; `daylightBufferMins` is defined in scoring config (default: 30)
- [ ] ISC-20: Windows starting before max(sunrise, earliestStart) are excluded; both defined in scoring config
- [ ] ISC-21: All scoring thresholds, weights, and timing values (daylightBufferMins, earliestStart, calm/breezy/rough wind thresholds, temp ideal band, rain probLow/High, mmLight/Heavy) live in a single `SCORING` config object — no magic numbers in scoring functions
- [ ] ISC-22: User's weight preferences from Settings override the default config at runtime
- [ ] ISC-23: Unit tests exist for `scoreHour()`, `scoreWindow()`, `rankWindows()` covering happy path and edge cases

### Results UI
- [ ] ISC-24: Best window displayed in a prominent card with score, time range, and one-line reason string
- [ ] ISC-25: All viable windows displayed in a scrollable ranked list below the best window card
- [ ] ISC-26: Each window row shows: score (numeric), score bar (colour-coded green/amber/red), start-end time, wind speed, gust, rain probability, feels-like temperature
- [ ] ISC-27: Score bar is green for score >= 75, amber for 50-74, red for < 50
- [ ] ISC-28: One-line reason string is generated by identifying the factor (rain/wind/temp) with the highest penalty contribution in the worst-scoring hour of the window; template: "{Factor} {descriptor} {time range}" e.g. "Gusty 11–13:00" / "Rain likely 10–11:00" / "Comfortable all morning"
- [ ] ISC-29: Tapping a window row expands to show hour-by-hour breakdown for that window

### Settings Screen
- [ ] ISC-30: Settings screen accessible from main screen (icon or menu)
- [ ] ISC-31: Units toggle: mph / kph (affects wind display and API request)
- [ ] ISC-32: Temperature toggle: °C / °F (affects temperature display and API request)
- [ ] ISC-33: Rain weight slider (0-100%, default 50%)
- [ ] ISC-34: Wind weight slider (0-100%, default 35%)
- [ ] ISC-35: Temperature weight slider (0-100%, default 15%)
- [ ] ISC-36: Weights normalise automatically so they always sum to 100% (auto-normalise on slider release; no manual total display needed)
- [ ] ISC-37: Round length preference stored and used as default on main screen
- [ ] ISC-38: Earliest tee time preference stored (default 07:00)
- [ ] ISC-39: Settings persist across app restarts (AsyncStorage)

### Play Store Readiness

## Architecture Sketch

```
┌─────────────────────────────────────────────────────┐
│  Expo (React Native + TypeScript)                     │
│  NativeWind for Tailwind-style CSS classes            │
│                                                       │
│  Screens:                                             │
│  ┌─────────────────┐  ┌─────────────────────────┐    │
│  │   HomeScreen    │  │     SettingsScreen       │    │
│  │  LocationInput  │  │   WeightSliders          │    │
│  │  DatePicker     │  │   UnitToggle             │    │
│  │  RoundLength    │  │   EarliestTee            │    │
│  │  ResultsList    │  │   RoundLength pref        │    │
│  └─────────────────┘  └─────────────────────────┘    │
│                                                       │
│  Services (pure functions, no side effects):          │
│  ┌────────────────────────────┐                       │
│  │  geocoding.ts              │ → Open-Meteo geocoding│
│  │  forecast.ts               │ → Open-Meteo forecast │
│  │  scoring.ts                │ → scoreHour/Window    │
│  │  settings.ts               │ → AsyncStorage r/w    │
│  └────────────────────────────┘                       │
│                                                       │
│  External APIs (no API key, free):                    │
│  • geocoding-api.open-meteo.com/v1/search             │
│  • api.open-meteo.com/v1/forecast                     │
│                                                       │
│  Device APIs:                                         │
│  • expo-location (GPS)                                │
│  • AsyncStorage (settings persistence)                │
└─────────────────────────────────────────────────────┘
         |
         ↓ EAS Build (cloud, free tier)
         |
         ↓ Google Play Store (AAB)
```

**Auth model:** None. No user accounts. No backend.

**Data model:** Ephemeral in-memory state (weather data) + AsyncStorage (user preferences only). No cloud database, no user data off-device.

**Integration points:** Two Open-Meteo APIs only. Both free, no key required.

**Deploy target:** Google Play Store (Android, API 34+). EAS Build generates the AAB.

## Security Posture

**Data classification:** Public. No personal data collected. No user accounts. No server.

**Threat model:**
- Only data touched: GPS coordinates (device-local, never transmitted except to Open-Meteo's public API), user preferences (device-local AsyncStorage).
- Open-Meteo receives lat/lng per forecast request — this is inherent to the service. Open-Meteo is a free public API with no user identification.
- No authentication surface, no injection surface, no server-side storage.

**OWASP Mobile Top 10 — v1 posture:**
| Risk | Relevance | Control |
|---|---|---|
| M1: Improper Credential Usage | No credentials in the app | N/A |
| M2: Inadequate Supply Chain Security | npm deps + Expo SDK | `npm audit` before each build; pin Expo SDK version |
| M3: Insecure Authentication | No auth | N/A |
| M4: Insufficient Input/Output Validation | Geocoding text input | Sanitise input before API call; display API errors gracefully |
| M5: Insecure Communication | HTTPS only | Open-Meteo is HTTPS; Expo enforces cleartext blocking by default |
| M6: Inadequate Privacy Controls | GPS access | Request only `whenInUse` location; privacy policy required by Play Store |
| M7: Insufficient Binary Protections | APK reverse-engineering | No secrets in binary; nothing to protect beyond public API URLs |
| M8: Security Misconfiguration | Android manifest permissions | Only INTERNET + LOCATION permissions; no WRITE_EXTERNAL_STORAGE etc. |
| M9: Insecure Data Storage | AsyncStorage for user prefs | No sensitive data — unit preferences are not sensitive |
| M10: Insufficient Cryptography | N/A | No data to encrypt |

**Privacy policy requirement:** Play Store mandates a privacy policy even if no data is collected. A one-page static HTML deployed to GitHub Pages stating "this app does not collect, store, or transmit personal data" is sufficient.

## Open Assumptions

1. **Course search UX**: Assumed a simple text → geocoding flow is sufficient for v1 (user types "Warwick", gets "Warwick, Warwickshire, UK" → lat/lng). If Duncan's courses are in small villages not in Open-Meteo's geocoding DB, this may not resolve. Mitigation in Slice 2: test against 3 real course towns before committing.

2. **EAS Build free tier**: Assumed 30 free builds/month is enough. If EAS queue waits are too long, local `eas build --local` is the fallback (requires Android Studio).

3. **Privacy policy domain**: Assumed a GitHub Pages URL is acceptable for Play Store. It is by policy, but must be verified during Slice 6.

4. **Scoring weight normalization**: Resolved — auto-normalize on slider release (ISC-36). No warning needed.

5. **Donate link policy**: External donate links (Ko-fi, BMaC) are technically in a grey area under Play Store policies. Framing as "support the app" (donation) rather than "unlock features" (payment for digital goods) is generally accepted. Verify current Play policy during Slice 6.

## External Critic Review

### Reviewer: Gemini 2.5 Pro (2026-06-14)

> This spec contains significant gaps and contradictions missed by the original agent.
>
> **Contradictions:**
> 1. **Scoring & Data:** The spec fetches `wind_gusts_10m` (ISC-10) but the `scoreHour()` function (ISC-16) only ingested `wind`, ignoring gusts — a critical factor for golf.
> 2. **Magic Numbers:** The scoring logic excluded windows using an undefined `buffer` (ISC-19), directly violating the "no magic numbers" principle (ISC-21).
> 3. **Settings UX:** Weight normalization was declared as a requirement (ISC-36) but also listed as an "Open Assumption," making the spec internally inconsistent.
>
> **Missing Decisions & Hand-waves:**
> - **Permissions:** The crucial user flow for GPS permission denial was missing. The app had no defined fallback, leading to a dead end.
> - **Reason String:** The logic for the "one-line reason string" (ISC-28) was completely hand-waved; determining a "dominant negative factor" is non-trivial product logic that remained undefined.
> - **Tee Times:** The 1-hour step for tee times (ISC-18) was an arbitrary constraint with no rationale.

**Resolution status:** All four issues resolved — ISC-16 updated to include gusts, ISC-19 uses named config field, ISC-28 defines reason-string generation logic, ISC-36/Open Assumptions inconsistency resolved, GPS denial fallback added as ISC-8b, 1-hour step rationale added to ISC-18.
